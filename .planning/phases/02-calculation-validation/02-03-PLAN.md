  <name>Task 2: Add write penalty validation tests (RAID 5/6)</name>
  <files>tests/engines/performance.spec.ts</files>
  <action>
    Expand performance.spec.ts with write penalty validation tests.

    Add describe block: "Performance Engine - RAID Write Penalties"

    RAID 5 write penalty tests (TEST-12):
    - Industry formula: 4x penalty for random writes
    - Reason: Read-Modify-Write cycle requires:
      - Read old data block
      - Read old parity block
      - Write new data block
      - Write new parity block
      = 4 I/O operations per logical write
    - Validate: writeIOPS = driveWriteIOPS / 4 * (driveCount - 1)
    - Test with multiple drive counts (4, 8, 12 drives)
    - Test with multiple drive types (HDD, SSD)

    RAID 6 write penalty tests (TEST-12):
    - Industry formula: 6x penalty for random writes
    - Reason: Double parity requires 6 I/O operations:
      - Read old data block
      - Read P parity block
      - Read Q parity block
      - Write new data block
      - Write new P parity
      - Write new Q parity
    - Validate: writeIOPS = driveWriteIOPS / 6 * (driveCount - 2)
    - Compare RAID 6 to RAID 5: RAID 6 should be slower for writes

    Document formulas with source attribution:
    - MassiveGRID: "Understanding RAID Write Penalties"
    - WintelGuy RAID Performance Calculator
    - Add URLs in test comments

    Sequential I/O tests:
    - Sequential writes do NOT suffer write penalty (full stripe writes)
    - Validate sequential bandwidth = aggregate drive bandwidth
    - Only random small-block I/O has penalty
  </action>
  <verify>
    npm test -- performance (write penalty tests pass)
    grep -c "write.*penalty.*4\\|write.*penalty.*6" tests/engines/performance.spec.ts (at least 4)

  </verify>

    Expand performance.spec.ts with bottleneck identification tests.

    Add describe block: "Performance Engine - Bottleneck Chain"

    Test bottleneck chain logic (TEST-05):
    - Performance limited by slowest link in chain:
      1. Media limit (drives aggregate IOPS/bandwidth)
      2. Controller limit (RAID controller max throughput)
      3. Bus limit (PCIe lanes, SAS backplane bandwidth)
      4. Network limit (NFS/iSCSI network speed)
    - Validate: actualPerformance = Math.min(media, controller, bus, network)

    Test scenarios:
    - Media-limited: 24x HDD (slow) with fast controller/network
      - Expect bottleneck = "media"
      - Actual IOPS = media IOPS, not controller IOPS
    - Controller-limited: 12x NVMe (fast) with slow RAID controller
      - Expect bottleneck = "controller"
      - Actual IOPS capped at controller limit
    - Network-limited: Fast array over 1GbE network
      - Expect bottleneck = "network"
      - Bandwidth capped at ~125MB/s (1Gbps)
    - Bus-limited: Many drives sharing single SAS backplane
      - Expect bottleneck = "bus"

    Property-based tests:
    - Increasing media performance cannot exceed controller/bus/network limits
    - Bottleneck correctly shifts when limits change
    - Adding drives cannot overcome non-media bottlenecks

    XFS stripe alignment tests:
    - Validate sunit/swidth calculations based on RAID chunk size
    - RAID 5: swidth = (N-1) * chunkSize
    - RAID 6: swidth = (N-2) * chunkSize
    - sunit = chunkSize in 512-byte blocks
  </action>
