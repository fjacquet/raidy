<objective>
Protect users from malicious URL hash manipulation by implementing runtime validation of deserialized state.

Purpose: Prevent injection of invalid values (negative counts, NaN, wrong enums) that could crash calculations or produce incorrect results. Client-side SPAs have no server-side validation, so URL deserialization is the security boundary.

Output: Zod-validated URL state loading with user notifications on failure, passing SEC-01, SEC-02, SEC-10 requirements.
</objective>

Create new file `src/utils/schemas.ts` with ConfigState validation schemas.

**Implementation requirements:**

1. Import Zod and config types:
```typescript
import { z } from 'zod'
import type { TopologyType } from '@/types/topology'
```

2. Define validation schemas with bounds:
   - DriveCount: integer, min 1, max 1000, must be finite (reject NaN/Infinity)
   - ServerCount: integer, min 1, max 100, optional
   - TopologyType: enum validation against ['standard', 'zfs', 's2d', 'ceph', 'vsan_esa', 'vsan_osa', 'powerflex', 'proprietary']
   - Drive capacity: positive number, finite
   - Performance values (IOPS, bandwidth): positive, finite

3. Create ConfigStateSchema matching Zustand store shape (refer to src/store/configStore.ts)

4. Export validation function:
```typescript
export function validateUrlState(data: unknown): ConfigState | null {
  const result = ConfigStateSchema.safeParse(data)
  if (result.success) {
    return result.data
  } else {
    console.error('Invalid URL state:', result.error.format())
    return null
  }
}
```

**Key constraints from research:**
- Use `.finite()` to reject NaN and Infinity
- Use `.int()` for count values
- Provide clear error messages for each field
- Handle optional fields with `.optional()`

Reference RESEARCH.md Pattern 1 (lines 65-101) for Zod schema examples.
